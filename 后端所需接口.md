# 后端所需接口（MVP 版）

> **一致性**：若本文与仓库内 `docs/api/*.md` 或 `backend/internal` 实现冲突，以 **`docs/api` + 源代码** 为准（本文保留早期规划与示例，未逐段随迭代更新）。

本文基于当前前端页面反向整理，可直接用于 `Gin + GORM` 开发。

覆盖页面/模块：
- Auth（邮箱登录注册 + 钱包签名绑定）
- Dashboard
- Accounts（X 账号管理）
- Automation（自动化配置）
- Activity（执行记录）
- Billing（试用/订阅/支付）
- 关键流程：钱包绑定、Twitter OAuth、执行日志系统、链上回调

---

## 1. 全局约定

### 1.1 Base URL 与版本
- Base URL：`/api/v1`

### 1.2 统一返回结构
所有接口统一：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

约定：
- `code = 0` 表示成功
- 非 0 表示业务错误（建议按模块分段编码）

### 1.3 鉴权方式
- `JWT access + refresh`
- Access Token 放在 Header：
  - `Authorization: Bearer <access_token>`
- Refresh Token 建议：
  - 放 HttpOnly Cookie（推荐），或 body 参数（二选一，建议 Cookie）

### 1.4 时间与分页
- 时间统一 ISO 8601（UTC），如 `2026-04-16T09:10:11Z`
- 分页 query：
  - `page`（默认 1）
  - `page_size`（默认 20，最大 100）

### 1.5 状态枚举（建议）
- 通用支付状态：`paid | pending | failed`
- 自动化模块：`post | reply | dm`
- 自动化运行态：`Running | Queued | Paused | Needs Review`
- 活动类型：`post | reply | dm`
- 活动状态：`success | review | failed`
- 账号状态：`connected | needs_reauth | disconnected`

## 2. Auth（登录/用户）

## 2.1 邮箱注册
- **POST** `/auth/register`
- 用途：邮箱创建账号
- 请求：

```json
{
  "email": "user@example.com",
  "password": "******"
}
```

- 返回重点：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "user": {
      "id": "u_xxx",
      "email": "user@example.com",
      "created_at": "2026-04-16T09:10:11Z"
    },
    "tokens": {
      "access_token": "jwt_access",
      "expires_in": 7200,
      "refresh_token": "jwt_refresh"
    }
  }
}
```

## 2.2 邮箱登录
- **POST** `/auth/login`
- 用途：邮箱密码登录
- 请求：

```json
{
  "email": "user@example.com",
  "password": "******"
}
```

- 返回：同注册返回结构（`user + tokens`）

## 2.3 刷新 token
- **POST** `/auth/refresh`
- 用途：续签 access token
- 请求（若 refresh 用 Cookie，可无 body）：

```json
{
  "refresh_token": "jwt_refresh"
}
```

- 返回重点：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "access_token": "new_access",
    "expires_in": 7200
  }
}
```

## 2.4 退出登录
- **POST** `/auth/logout`
- 用途：注销 refresh token（服务端黑名单/版本号机制）

## 2.5 当前用户信息
- **GET** `/users/me`
- 用途：前端初始化用户态、会员态、绑定状态
- 返回重点：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "id": "u_xxx",
    "email": "user@example.com",
    "display_name": "Octo User",
    "wallet_address": "0xabc...",
    "plan": "free_trial",
    "trial_days_left": 5,
    "subscription_status": "active"
  }
}
```

## 3. 钱包绑定（connect + sign + bind）

> 当前前端已可拿到钱包地址，这里建议后端实现签名校验闭环，保证地址归属。

## 3.1 获取签名 challenge
- **POST** `/wallet/challenge`
- 用途：后端签发一次性 challenge（含 nonce、过期时间）
- 请求：

```json
{
  "address": "0xabc...",
  "chain_id": 1
}
```

- 返回重点：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "challenge_id": "wc_xxx",
    "message": "Sign this message to bind wallet...\nNonce: xxx",
    "nonce": "xxx",
    "expired_at": "2026-04-16T09:20:11Z"
  }
}
```

## 3.2 校验签名并绑定钱包
- **POST** `/wallet/bind`
- 用途：验证签名成功后，把钱包绑定到当前登录用户
- 请求：

```json
{
  "challenge_id": "wc_xxx",
  "address": "0xabc...",
  "signature": "0xsign...",
  "chain_id": 1
}
```

- 返回重点：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "wallet_address": "0xabc...",
    "bound_at": "2026-04-16T09:12:11Z"
  }
}
```

## 3.3 解绑钱包
- **DELETE** `/wallet/bind`
- 用途：移除当前用户钱包绑定

## 3.4 钱包登录（可选但建议）
- **POST** `/auth/wallet/login`
- 用途：仅靠钱包签名登录并签发 JWT（后续可支持“仅钱包用户”）
- 请求：同 `/wallet/bind` 的签名字段
- 返回：`user + tokens`

## 5. Accounts（X 账号管理）

## 5.1 获取已绑定账号列表
- **GET** `/accounts`
- 用途：Accounts 页面列表
- 返回重点：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "id": "acc_xxx",
        "avatar_url": "https://...",
        "username": "octo_team",

        "status": "connected",
        "last_synced_at": "2026-04-16T09:10:11Z",
        "followers": "12.6K"
      }
    ]
  }
}
```

## 5.2 获取单账号详情
- **GET** `/accounts/{account_id}`
- 用途：账号详情页/管理弹窗详情

## 5.3 断开账号
- **DELETE** `/accounts/{account_id}`
- 用途：解绑 X 账号

## 5.4 重新授权账号
- **POST** `/accounts/{account_id}/reauthorize`
- 用途：生成新的 OAuth 授权地址
- 返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "auth_url": "https://twitter.com/i/oauth2/authorize?...",
    "state": "oauth_state_xxx"
  }
}
```

## 5.5 发起新增账号 OAuth
- **POST** `/accounts/oauth/x/start`
- 用途：新增 X 账号时获取授权地址
- 返回：同上（`auth_url + state`）

## 5.6 OAuth 回调落地
- **GET** `/accounts/oauth/x/callback`
- 用途：服务端接收 Twitter 回调，换 token，落库账号绑定关系
- query：`code`, `state`
- 返回：建议重定向前端 `/accounts?oauth=success|failed`

## 8.6 支付记录列表
- **GET** `/billing/payments`
- **当前仓库未实现该路由**；前端若展示历史，可依赖订单轮询/本地状态或后续新增列表接口。

## 8.7 链上回调/webhook（关键）
- **POST** `/billing/webhooks/onchain`（**无 Bearer**；Header **`X-Billing-Webhook-Secret`** 与配置 `billing.webhook_secret` 一致）
- 用途：提交链上交易哈希以触发服务端 **BEP20** 转账校验与订单确认
- 请求（实际实现）：

```json
{
  "order_id": "12",
  "network": "BEP20",
  "tx_hash": "0xabc..."
}
```

- 服务端处理要点：
  - 校验 Header 密钥
  - 链上校验金额/收款/合约；`billing_chain_txs` 对 `chain_id`+`tx_hash` 幂等
  - 更新订单状态并延长用户订阅

## 9. 关键流程设计（流程级）

## 9.1 钱包绑定流程
1. 前端拿到钱包地址后请求 `/wallet/challenge`
2. 钱包签名 challenge message
3. 前端提交 `/wallet/bind`
4. 后端验签成功后绑定 address 到当前 user
5. `/users/me` 可返回 `wallet_address`

## 9.2 Twitter OAuth 绑定流程
1. 前端请求 `/accounts/oauth/x/start`
2. 跳转 `auth_url`
3. Twitter 回调 `/accounts/oauth/x/callback`
4. 后端换 token、拉取用户信息、写入账户绑定
5. 重定向前端 `/accounts` 并展示结果

## 9.3 自动化执行与日志流程
1. 用户通过 `/automations/{type}` 更新配置
2. 调度器读取配置执行任务：**发帖**（`post_scheduler`）、**自动回复**（`auto_reply_job`）；**自动私信尚未实现**
3. 每次执行写 `activity_logs`
4. `/activities` 提供筛选查询
5. `/automations/runtime-status` 提供 **基于 DB 的** 队列估计、失败数、待审核数、最近成功时间（见 `docs/api/automation.md`）

## 9.4 试用/订阅状态流转
1. 新用户注册 -> `free_trial`，写 `trial_end_at`
2. 创建支付单 `/billing/orders`
3. 支付后 webhook 回调 `/billing/webhooks/onchain`
4. 后端确认后开通 `basic`，写 `subscription_end_at`
5. Dashboard/Billing 通过 `/billing/subscription` 与 `/dashboard/overview` 展示状态

## 10. MVP 必须实现 vs 建议预留

### 必须实现（前端联调必需）
- Auth：`/auth/register`, `/auth/login`, `/auth/refresh`, `/users/me`
- Wallet：`/wallet/challenge`, `/wallet/bind`, `/wallet/bind (DELETE)`
- Dashboard：`/dashboard/overview`
- Accounts：`/accounts`, `/accounts/oauth/x/start`, `/accounts/oauth/x/callback`, `/accounts/{id} DELETE`
- Automation：`/automations`, `/automations/{type}`, `/automations/{type}/toggle`, `/automations/runtime-status`

- Billing：`/billing/subscription`, `/billing/plans`, `/billing/payment-methods`, `/billing/orders`, `/billing/orders/{id}`, `/billing/webhooks/onchain`（**无** `/billing/payments` 路由）

- `/activities/stats`
- `/accounts/{id}` 详情增强字段
- `/auth/wallet/login`（纯钱包登录）
- Billing 对账后台接口（admin）

## 11. 错误码建议（示例）

- `0`：成功
- `1001`：参数错误
- `1002`：未认证/Token 失效
- `1003`：无权限
- `2001`：用户不存在
- `2002`：邮箱或密码错误
- `2101`：wallet challenge 失效
- `2102`：钱包签名校验失败
- `2201`：Twitter OAuth state 非法
- `2301`：自动化配置非法
- `2401`：活动记录查询失败
- `2501`：订单不存在
- `2502`：支付金额不匹配
- `2503`：webhook 签名校验失败
- `2504`：webhook 重放/重复事件

## 12. 后端数据模型设计（MVP）

本节仅定义工程可落地的数据模型结构与关系，不包含业务代码与 SQL。

### 12.1 现有模型与演进策略

当前后端已有模型：`User`、`TwitterAccount`、`Agent`、`Task`、`Post`。  
建议：
- `User` 保留并扩展字段；
- `TwitterAccount` 升级为 `XAccount`（可兼容原表名渐进迁移）；
- `Agent/Task/Post` 暂不删除，避免影响已有代码路径；
- 新增本期 MVP 核心模型，优先服务现有前端页面联调。

### 12.2 核心模型（MVP 必须）

#### 1) User
- 核心字段：
  - `id` (PK)
  - `email` (unique)
  - `password_hash`
  - `display_name`
  - `status` (`active|disabled`)
  - `created_at`, `updated_at`
- 说明：邮箱登录主实体；钱包信息不要直接塞在 User 表。

#### 2) WalletChallenge
- 核心字段：
  - `id` (PK)
  - `challenge_id` (unique，对外使用)
  - `address`
  - `chain_id`
  - `nonce` (unique)
  - `message`
  - `expired_at`
  - `used_at` (nullable)
  - `created_at`
- 说明：用于 `challenge -> sign -> bind` 防重放。

#### 3) UserWallet
- 核心字段：
  - `id` (PK)
  - `user_id` (index)
  - `address` (index)
  - `chain_id`
  - `is_primary` (bool)
  - `bound_at`
  - `unbound_at` (nullable)
  - `created_at`, `updated_at`
- 关键约束：
  - `(address, chain_id)` unique（同链同地址只绑定一个用户）

#### 4) XAccount
- 核心字段：
  - `id` (PK)
  - `user_id` (index)
  - `platform`（默认 `x`）
  - `x_user_id` (unique)
  - `username`
  - `display_name`
  - `avatar_url`

### 12.3 关系设计

- `User 1:N UserWallet`
- `User 1:N WalletChallenge`
- `User 1:N XAccount`
- `User 1:N AutomationConfig`
- `User 1:N ActivityLog`
- `User 1:1 Subscription`
- `User 1:N BillingOrder`
- `User 1:N PaymentRecord`
- `BillingOrder 1:N PaymentRecord`
- `XAccount 1:N ActivityLog`

关系要点：
- `Subscription` 建议按用户唯一，聚焦“当前态”；
- 自动化配置按 `module_type` 一人一条；
- 支付流水与订单分离，支持重试与链上多事件。

### 12.5 GORM AutoMigrate 组织思路

- 模型文件建议统一放在 `backend/internal/model/`。
- `AutoMigrate` 注册顺序按依赖：
  1. `User`
  2. `WalletChallenge`, `UserWallet`, `XAccount`
  3. `AutomationConfig`, `ActivityLog`
  4. `Subscription`, `BillingOrder`, `PaymentRecord`
  5. （可选）`WebhookEvent`, `RefreshToken`, `OAuthState`

Tag 约定建议：
- 主键：`gorm:"primaryKey"`
- 唯一：`gorm:"uniqueIndex"`
- 组合唯一：`gorm:"uniqueIndex:uk_user_module"`
- 常用查询字段：`index`
- JSON 字段：`type:json`
- 状态/枚举字段：`size + index`

### 12.6 防过度设计原则

- 先覆盖前端当前页面字段，保证联调闭环；
- 扩展信息放入 `meta_json`，减少早期表结构膨胀；
- 暂不做分表、归档、复杂事件溯源，留待数据量上来后演进。
