# 第一批接口测试材料（Auth + Wallet）

适用范围：
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `GET /api/v1/users/me`
- `POST /api/v1/wallet/challenge`
- `POST /api/v1/wallet/bind`
- `DELETE /api/v1/wallet/bind`

---

## 测试前置条件

1. 启动 MySQL，确保 `backend/configs/config.local.yaml` 指向可用库。  
2. 启动后端：
   - `make api-local`
3. 默认 Base URL（按本地配置）：`http://localhost:10001/api/v1`
4. 准备测试变量（Postman/Apifox 环境变量）：
   - `baseUrl`: `http://localhost:10001/api/v1`
   - `email`: `qa_auth_wallet_001@test.com`
   - `password`: `123456`
   - `name`: `qa-user`
   - `accessToken`: 空（登录后写入）
   - `refreshToken`: 空（登录后写入）
   - `challengeId`: 空（challenge 后写入）
   - `walletAddress`: `0xabc123`
   - `chainId`: `1`
   - `signature`: `0xdeadbeef`

5. 前端联调前置（新增）：
   - 前端已切换为真实后端调用（非 mock）。
   - 建议在 `frontend/.env.local` 配置：
     - `NEXT_PUBLIC_API_BASE_URL=http://localhost:10001/api/v1`
   - 启动前端后，登录/注册/发送验证码/钱包绑定将直接请求上述接口。

> 说明：当前实现成功响应结构为 `{code:0,message:"ok",data:{...}}`。  
> 失败场景的 `code` 目前等于 HTTP status（例如 400/401）。

## 一、正常流程测试

## 1) 发送验证码 `POST /auth/email-code/send`

### Postman / Apifox 请求
- Method: `POST`
- URL: `{{baseUrl}}/auth/email-code/send`
- Body(JSON):
```json
{
  "email": "{{email}}",
  "purpose": "register"
}
```

### curl
```bash
curl -X POST "http://localhost:10001/api/v1/auth/email-code/send" \
  -H "Content-Type: application/json" \
  -d '{"email":"qa_auth_wallet_001@test.com","purpose":"register"}'
```

### 预期返回
- HTTP `200`
- `code = 0`
- `data.expires_in = 600`
- 本地环境下返回 `data.code`（请保存用于后续注册）

## 二、异常测试

## Auth 异常
- `/auth/email-code/send` 频繁发送（60秒内重复请求）
  - 预期：HTTP `400`，提示稍后重试
- `/auth/email-code/send` 传不支持的 `purpose`
  - 预期：HTTP `400`，`message` 包含 `invalid email code purpose`
- `/auth/register` 不传 `verification_code`
  - 预期：HTTP `400`（参数校验失败）
- 注册参数不合法（邮箱格式错/密码<6）
  - 预期：HTTP `400`
- 重复注册同邮箱
  - 预期：HTTP `400`，`message` 包含 `email already exists`
- 登录密码错误
  - 预期：HTTP `401`，`message` 包含 `invalid email or password`
- refresh token 非法
  - 预期：HTTP `401`，`message` 包含 `invalid refresh token`
- `GET /users/me` 缺 Authorization
  - 预期：HTTP `401`，`message` 为 `missing bearer token`
- `GET /users/me` token 无效
  - 预期：HTTP `401`，`message` 为 `invalid token`

## Wallet 异常
- `/wallet/challenge` 缺 `address` 或 `chain_id`
  - 预期：HTTP `400`
- `/wallet/bind` `signature` 非 `0x` 开头
  - 预期：HTTP `400`，`message` 为 `invalid signature format`
- `/wallet/bind` 使用不存在或过期 challenge
  - 预期：HTTP `400`，`message` 为 `challenge is invalid or expired`
- `/wallet/bind` challenge 与 address/chain_id 不匹配
  - 预期：HTTP `400`，`message` 为 `challenge mismatch`
- `/wallet/bind` 重复使用同一 challenge
  - 预期：第二次 HTTP `400`（因 challenge 已 used）
- `/wallet/*` 未带 token
  - 预期：HTTP `401`

## 三、数据库验证

## 需要验证的表
- `users`
- `email_verification_codes`
- `wallet_challenges`
- `user_wallets`

## 关键字段验证点

### A. 注册后（`users`）
- `email`：等于注册邮箱
- `password_hash`：不等于明文密码
- `display_name`：等于请求 name（空时默认 `user`）
- `status`：`active`

### B. challenge 后（`wallet_challenges`）
- `challenge_id`：唯一，且与响应一致
- `user_id`：等于当前登录用户 ID
- `address`：小写存储
- `chain_id`：正确
- `nonce`：唯一
- `expired_at`：约 5 分钟后
- `used_at`：初始为 `NULL`

### C. bind 后
- `wallet_challenges.used_at`：从 `NULL` 变为非空
- `user_wallets`：
  - 有对应 `user_id + address + chain_id`
  - `is_primary = true`
  - `unbound_at = NULL`

### D. unbind 后
- `user_wallets.unbound_at`：变为非空
- `user_wallets.is_primary = false`

### E. email code 校验
- `email_verification_codes` 发送后存在新记录：
  - `email`、`purpose`、`code` 正确
  - `expired_at` 约等于当前时间 + 10 分钟
  - `used_at = NULL`
- 验证成功后，该条验证码记录被删除（一次性使用）

## 数据库核验 SQL 示例（可在 MySQL 客户端执行）
```sql
SELECT id,email,password_hash,display_name,status,created_at FROM users WHERE email='qa_auth_wallet_001@test.com';
SELECT id,email,purpose,code,expired_at,used_at,created_at FROM email_verification_codes WHERE email='qa_auth_wallet_001@test.com' ORDER BY id DESC LIMIT 10;
SELECT id,challenge_id,user_id,address,chain_id,nonce,expired_at,used_at FROM wallet_challenges ORDER BY id DESC LIMIT 5;
SELECT id,user_id,address,chain_id,is_primary,bound_at,unbound_at FROM user_wallets WHERE user_id=<user_id> ORDER BY id DESC;
```

## 补充说明（联调时常见问题）
- 当前 `refresh` 为简化方案，`refresh_token` 与 access token 同签发体系，仅用于 MVP 联调。
- 当前钱包验签是 MVP 简化逻辑（非链上完整 EIP 验签），所以测试时 `signature` 传 `0x...` 即可。
- 如果你看到失败响应 `code` 为 400/401，这是当前 `response.Fail` 设计（`code=HTTP状态码`），属于当前实现预期行为。
- 前端请求层会自动附带 `Authorization: Bearer <access_token>`，401 时会清理本地登录态。
- 前端钱包流程已统一为真实链路：`POST /wallet/challenge -> signMessage -> POST /wallet/bind`。
